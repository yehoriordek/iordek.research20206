# Документація проекту для Claude Code

> **Цей файл містить**: загальну архітектуру проекту, ролі Claude Code, потоки даних, процес розробки та принципи комунікації з користувачем.
> **Цей файл не містить і ніколи не повинен містити**: детальну структуру полів анкети, ID Airtable, можливі значення полів, технічні деталі `index.html` чи покрокові інструкції AI-обробки фото. Така інформація описується **тільки** у відповідному файлі нижчого рівня (`data-structure.md`, `image-to-airtable.md`) — сюди вона не копіюється, навіть частково. Тут лише короткі посилання на ці файли.

## Про користувача

**Егор** — психолог без досвіду програмування. Відповіді мають бути адаптовані для людини без технічного бекграунду та надаватися **виключно українською мовою**.

## Архітектура проекту

### Мета
Розробка веб-додатку для збору анкетних даних з подальшим збереженням у базі даних Airtable.

### Основні компоненти

#### 1. **Markdown-файл структури даних** (`data-structure.md`)
- Основний файл проекту
- Містить перелік усіх полів форми
- Описує тип кожного поля, його призначення та можливі значення (для перечислюваних типів)
- Служить джерелом правди для всіх інших компонентів

#### 2. **HTML-форма** (`index.html`)
- Standalone веб-сторінка
- Може відкриватися локально з браузера
- Розміщена на GitHub.io
- Містить дві режими:
  - **Режим заповнення**: для введення даних через форму
  - **Режим друку**: переключач у початку сторінки для превращення форми в друковану версію
- Друкована версія — це чистий паперовий бланк для заповнення олівцем від руки: текст підказок не друкується, а випадаючі списки мають перетворюватися на варіанти для позначення галочкою

#### 3. **AI Skill-файл** (`image-to-airtable.md`)
- Інструкції для моделі Claude для обробки фотографій заповнених від руки анкет
- Містить:
  - Опис структури даних
  - Послідовність кроків для розпізнавання та введення даних
  - Контроль якості з уточнюючими запитаннями
  - Вказівки для запису в Airtable через MCP

#### 4. **Додаткові документи** (за необхідністю)
- `ARCHITECTURE.md` — описання архітектури та взаємодії компонентів
- `UI-GUIDELINES.md` — вказівки щодо інтерфейсу
- `AI-INTERACTION.md` — описання взаємодії користувача з AI при сканування фото
- `CHANGELOG.md` — журнал змін (українська мова)

## Потоки даних

### Спосіб 1: Веб-форма
```
Користувач заповнює HTML-форму → Дані → Airtable (MCP-підключення)
```

### Спосіб 2: Фотографія анкети від руки
```
Користувач робить фото → Надсилає Claude Code → 
AI розпізнає дані → Уточнює через питання → 
Записує в Airtable (MCP)
```

### Спосіб 3: Друкована форма
```
Користувач друкує HTML (режим друку) → Заповнює від руки →
Робить фото → Надсилає Claude Code → Обробка як у Способі 2
```

## Поточний стан проекту (станом на v0.37.0)

Підключення до Airtable налаштоване та перевірене. Всі форми функціонують. Головне меню (`index.html`) — стартова точка для роботи з системою. Повна структура полів, ID Airtable та відомі неузгодженості — у `data-structure.md`.

### `patient.html` — технічні деталі
- Standalone HTML5 + ванільний JS, без збірки та залежностей
- `localStorage` зберігає введені дані між сесіями (ключ `patientFormData`)
- Відправка форми (`submit`) запитує пароль і записує дані напряму в Airtable через REST API (ключ зберігається у формі в зашифрованому вигляді — деталі в `.claude/skills/airtable-key-encryption/SKILL.md`); запис через Claude Code / MCP залишається для обробки фотографій анкет
- Перемикач "Режим друку" додає клас `print-layout` до `<body>` і викликає `window.print()`
- У режимі друку плейсхолдери не друкуються, а випадаючі списки показуються як набір квадратів для позначення від руки

## Принципи розробки

### Архітектура
- **Не ускладнюй**: розпочинай з простих рішень
- **Скептичність до змін**: архітектуру мінятись тільки після глибокого обговорення
- **Контрольованість**: будь-які зміни мають бути протестовані та документовані

### Прийняття рішень — дій самостійно
- Поточна архітектура зафіксована: одна людина-агент (Claude Code) через MCP редагує `data-structure.md`, `index.html`, `image-to-airtable.md` та структуру таблиці в Airtable. Це усталена схема — **не пропонуй користувачу її змінювати чи обирати альтернативи**.
- Для звичайних завдань (додати/змінити/прибрати поле анкети, виправити текст, виправити баг у формі) **не задавай користувачу технічні або архітектурні питання**. Сам визнач найпростіший спосіб у рамках поточної архітектури і реалізуй його.
- Якщо щось перестало працювати (GitHub-з'єднання, MCP-сервер Airtable, помилка при пуші тощо) — це **несправність, яку треба полагодити**, а не привід шукати обхідний шлях чи альтернативну архітектуру. Розберись у причині та віднови роботу так, як було задумано.
- Уточнюючі питання користувачу став **тільки** тоді, коли йдеться про зміст анкети (нові поля, формулювання, варіанти відповідей) — тобто про те, що реально залежить від його фахового рішення як психолога.
- Будь-який елемент форми треба одразу продумувати в двох режимах: для екрана і для друку. Друкований режим завжди має залишатися зручним для заповнення олівцем живою людиною на білому аркуші: без текстів-підказок усередині полів, а елементи вибору — у вигляді варіантів, де можна поставити позначку від руки.

### Коли можна самостійно комітити, пушити та створювати PR
- Якщо повідомлення користувача — **питання, прохання пояснити, чи обговорення варіантів** ("як це працює", "чому", "де зберігається", "що краще") — відповідай тільки в чаті. Нічого не створюй, не комітуй і не пуш, навіть якщо відповідь логічно тягне за собою якийсь файл чи код. Можеш запропонувати це зробити — і чекай підтвердження.
- Якщо повідомлення містить **чітку дію-наказ** ("додай", "зроби", "виправ", "впровадь", "оновити X") і завдання технічно завершене (відомі всі дані, потрібні для реалізації) — реалізуй, закомітуй і запуш без додаткових запитань.
- Якщо завдання складається з кількох кроків, а частина вхідних даних ще відсутня (наприклад, чекаємо на ключ/пароль від користувача) — **не створюй "часткові" PR** під ще нефункціональну фічу. Дочекайся повного набору даних, або явно запитай одним реченням, чи зробити поетапно.
- Самостійний **merge PR** дозволений лише коли: (а) зміна реалізує саме те, про що користувач явно попросив, (б) вона завершена і не є проміжним/допоміжним артефактом, (в) користувач не натякав, що хоче ще щось перевірити чи обговорити перед публікацією.
- У разі сумніву — краще одним коротким реченням запитати "зробити це зараз?", ніж одразу комітити.

### Комунікація
- **Українська мова**: всі пояснення тільки українською
- **Адаптація**: не використовуй технічні терміни, ID, гілки, коміти, URL-адреси репозиторію тощо — користувач не програміст і це йому не потрібно
- **Без зайвого ентузіазму**: не пиши "✅ Готово!", "🎉", "🚀" та подібні емоційні позначки. Повідомляй спокійно і по суті: що зроблено і чи є щось, що потрібно перевірити користувачу
- **Стислість**: коротке повідомлення про суть зміни (1-3 речення). Не перелічуй технічні кроки (коміт, pull request, push) — для користувача це "я зберіг та опублікував зміни"
- **Прозорість там, де це важливо**: якщо щось не вдалося і користувачу треба щось зробити (наприклад, перевірити форму вручну) — скажи прямо, без техдеталей

### Якість

- Тестуй HTML-форму перед pushем

### Документація
- Кожен факт (поле анкети, ID Airtable, значення, технічна деталь) описується **один раз**, у файлі, до якого він належить (`data-structure.md`, `image-to-airtable.md` тощо)
- У `CLAUDE.md` такі деталі **не повторюються** — лише короткі посилання на відповідний файл
- На початку кожного md-файлу має бути короткий блок "що тут є / чого тут нема", щоб було зрозуміло, куди вносити зміни
- Якщо при роботі знаходиш дублювання чи суперечність між файлами — виправ її одразу, перенісши деталі в правильний файл

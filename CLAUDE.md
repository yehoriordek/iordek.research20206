## Архітектура проекту

### Мета
Розробка веб-додатку для збору анкетних даних з подальшим збереженням у базі даних Airtable.

### Основні компоненти

#### 1. **Markdown-файл структури даних** (`data-structure.md`)
- Основний файл проекту
- Містить перелік усіх полів форми
- Описує тип кожного поля, його призначення та можливі значення (для перечислюваних типів)
- Служить джерелом правди для всіх інших компонентів

#### 2. **HTML-форма** (`index.html`)
- Standalone веб-сторінка
- Може відкриватися локально з браузера
- Розміщена на GitHub.io
- Містить дві режими:
  - **Режим заповнення**: для введення даних через форму
  - **Режим друку**: переключач у початку сторінки для превращення форми в друковану версію
- Друкована версія — це чистий паперовий бланк для заповнення олівцем від руки: текст підказок не друкується, а випадаючі списки мають перетворюватися на варіанти для позначення галочкою

#### 3. **AI Skill-файл** (`image-to-airtable.md`)
- Інструкції для моделі Claude для обробки фотографій заповнених від руки анкет
- Містить:
  - Опис структури даних
  - Послідовність кроків для розпізнавання та введення даних
  - Контроль якості з уточнюючими запитаннями
  - Вказівки для запису в Airtable через MCP

#### 4. **Додаткові документи** (за необхідністю)
- `ARCHITECTURE.md` — описання архітектури та взаємодії компонентів
- `UI-GUIDELINES.md` — вказівки щодо інтерфейсу
- `AI-INTERACTION.md` — описання взаємодії користувача з AI при сканування фото
- `CHANGELOG.md` — журнал змін (українська мова)

## Потоки даних

### Спосіб 1: Веб-форма
```
Користувач заповнює HTML-форму → Дані → Airtable (MCP-підключення)
```

### Спосіб 2: Фотографія анкети від руки
```
Користувач робить фото → Надсилає Claude Code → 
AI розпізнає дані → Уточнює через питання → 
Записує в Airtable (MCP)
```

### Спосіб 3: Друкована форма
```
Користувач друкує HTML (режим друку) → Заповнює від руки →
Робить фото → Надсилає Claude Code → Обробка як у Способі 2
```

## Поточний стан проекту (станом на v0.2.0)

### Підключення до Airtable
- **База**: `iordek.research2026` (ID: `appRtxXIvhSWdb6sJ`)
- **Таблиця**: Учасники (ID: `tblMV2RrZGmcGAdqz`)
- MCP-підключення налаштоване та перевірене (`mcp__Airtable__*`), готове до запису записів.

### Поля анкети (синхронізовані в `data-structure.md`, `index.html`, `image-to-airtable.md`)
1. **Кодовий нікнейм пацієнта** (`patient_nickname`)
   - Тип: текст (singleLineText), обов'язкове
   - Airtable field ID: `fldyUD3kFFZNJmbPF` (назва в Airtable: "Ім'я")
2. **Стать пацієнта** (`patient_gender`)
   - Тип: вибір (singleSelect), обов'язкове
   - Airtable field ID: `fldtIgwWqErKM3Jz0` (назва в Airtable: "Стать")
   - Значення в `index.html`/HTML-формі: `M`, `F`, `O`, `NA`
   - Значення в Airtable (`data-structure.md`): `Чоловік`, `Жінка`

   ⚠️ **Невідповідність значень статі**: HTML-форма використовує коди `M/F/O/NA`, а Airtable очікує текстові значення `Чоловік`/`Жінка` (без варіантів для O/NA). При наступних змінах структури даних варто узгодити це — або додати відповідні опції в Airtable, або привести значення форми до варіантів Airtable.

### `index.html` — технічні деталі
- Standalone HTML5 + ванільний JS, без збірки та залежностей
- `localStorage` зберігає введені дані між сесіями (ключ `patientFormData`)
- Відправка форми (`submit`) поки що лише логує дані в консоль — інтеграція з Airtable через MCP ще не реалізована безпосередньо у формі (запис відбувається через Claude Code / MCP при обробці фото)
- Перемикач "Режим друку" додає клас `print-layout` до `<body>` і викликає `window.print()`
- У режимі друку плейсхолдери не друкуються, а випадаючі списки показуються як набір квадратів для позначення від руки

## Принципи розробки

### Архітектура
- **Не ускладнюй**: розпочинай з простих рішень
- **Скептичність до змін**: архітектуру мінятись тільки після глибокого обговорення
- **Контрольованість**: будь-які зміни мають бути протестовані та документовані

### Прийняття рішень — дій самостійно
- Поточна архітектура зафіксована: одна людина-агент (Claude Code) через MCP редагує `data-structure.md`, `index.html`, `image-to-airtable.md` та структуру таблиці в Airtable. Це усталена схема — **не пропонуй користувачу її змінювати чи обирати альтернативи**.
- Для звичайних завдань (додати/змінити/прибрати поле анкети, виправити текст, виправити баг у формі) **не задавай користувачу технічні або архітектурні питання**. Сам визнач найпростіший спосіб у рамках поточної архітектури і реалізуй його.
- Якщо щось перестало працювати (GitHub-з'єднання, MCP-сервер Airtable, помилка при пуші тощо) — це **несправність, яку треба полагодити**, а не привід шукати обхідний шлях чи альтернативну архітектуру. Розберись у причині та віднови роботу так, як було задумано.
- Уточнюючі питання користувачу став **тільки** тоді, коли йдеться про зміст анкети (нові поля, формулювання, варіанти відповідей) — тобто про те, що реально залежить від його фахового рішення як психолога.
- Будь-який елемент форми треба одразу продумувати в двох режимах: для екрана і для друку. Друкований режим завжди має залишатися зручним для заповнення олівцем живою людиною на білому аркуші: без текстів-підказок усередині полів, а елементи вибору — у вигляді варіантів, де можна поставити позначку від руки.
